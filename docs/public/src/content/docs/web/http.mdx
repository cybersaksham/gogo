Gogo HTTP code belongs in project and app URL files:

- project root routes in `<project>/urls.go`
- app routes in `apps/<app>/urls.go`
- middleware wiring in project settings or `<project>/middleware.go`

## Capabilities

| Area | Includes |
| --- | --- |
| Routing | route registration, path patterns, URL reversing, app route wiring |
| Requests and responses | request wrappers, response helpers, redirects, error responses |
| Middleware | ordered middleware chains for security, sessions, CSRF, auth, messages, and custom behavior |
| Views | function views, generic views, decorators, and shared handler helpers |
| Errors | centralized error handling and consistent HTTP responses |

## Framework And Raw Handlers

Use `router.Handle` for Gogo framework views that return `gogohttp.Response`.
Use `router.HandleHTTP` when mounting an existing `net/http.Handler` directly.
Raw handlers receive the original `http.ResponseWriter` and `*http.Request`, so
streaming, flushing, hijacking, upgrades, and file-serving behavior are
preserved. Path parameters are also available through `r.PathValue("name")`.

`gogohttp.FromHandler` remains useful for adapting simple non-streaming handlers
into framework views. Use `HandleHTTP` for handlers that depend on the real
writer, such as Server-Sent Events, WebSockets, response flushing, or
`http.ResponseController`.

Keep route handlers thin. Put reusable business logic in app services so admin,
API, tasks, and tests can reuse the same code.

## Access Logs

`gogo.http.AccessLogMiddleware` writes structured JSON with method, path, host,
remote address, status, duration, request ID, and `route_name` when the request
matches either a framework route or a raw `net/http.Handler` route.

# Upgrade Process

Use this process when upgrading a client project to a newer Gogo version. The
goal is to upgrade framework code, generated project conventions, migrations,
and runtime services without changing production data until the upgrade has
been reviewed.

## Before You Upgrade

1. Read `CHANGELOG.md` for the target version.
2. Read `docs/code/operations/compatibility.md` for supported Go, database, broker,
   and generated project contracts.
3. Confirm the target version supports the project's database, queue broker,
   result backend, and Go version.
4. Create a database backup and verify restore instructions before running
   migrations in production.
5. Run the current project test suite before changing dependencies.
6. Capture current migration state with `gogo showmigrations`.
7. Capture the current production settings values, excluding secrets.

Do not upgrade production directly from an untested local checkout. Test the
upgrade in a staging environment with production-like settings and data volume.

## Upgrade The Framework Version

In a generated project, update the module requirement:

```bash
go get github.com/cybersaksham/gogo@v0.1.0
go mod tidy
go test ./...
```

When testing an unreleased local framework checkout, use a temporary `replace`
directive and remove it before publishing the application:

```bash
go mod edit -replace github.com/cybersaksham/gogo=/absolute/path/to/gogo
go mod tidy
go test ./...
```

After dependency updates, confirm the CLI version used by automation:

```bash
gogo version
```

The project should use the same `gogo` binary version as the module version in
`go.mod`.

## Migration Safety Checks

Run migration checks before applying changes:

```bash
gogo makemigrations --check --dry-run
gogo migrate --plan
gogo showmigrations
```

Review generated migration files before they are committed. Check that every
operation is intentional, reversible where the project requires rollback, and
safe for the expected table size.

For production:

- Take and verify a database backup.
- Run migrations in staging against a recent production copy.
- Review raw SQL generated by `gogo sqlmigrate` for risky operations.
- Prefer additive migrations for large tables.
- Split destructive operations into separate releases.
- Keep the old application version available until migrations and smoke tests
  pass.
- Never edit migration files that have already been applied in any shared
  environment.

If `gogo migrate --plan` shows unexpected apps, dependencies, or operation
order, stop the upgrade and inspect the migration graph before applying.

## Breaking Changes

For pre-`v1.0.0` releases, minor versions may include breaking changes. For
`v1.0.0` and later, breaking changes require a major version.

Handle breaking changes with this order:

1. Read the changelog entry and upgrade notes for the target version.
2. Apply required code changes in a branch before changing production settings.
3. Update generated project files only after comparing local customizations.
4. Run tests and migration dry runs.
5. Deploy to staging.
6. Run smoke tests for admin, auth, APIs, queues, static files, media files,
   and migrations.
7. Deploy production with rollback instructions ready.

Do not combine a framework major upgrade with unrelated application features.

## Generated Template Drift

Generated projects are owned by the application after creation, so framework
upgrades do not automatically rewrite local files. Compare new generated
templates with the existing project when release notes mention generator
changes.

Create a temporary project with the new framework version:

```bash
tmpdir="$(mktemp -d)"
gogo startproject compare "$tmpdir/compare"
gogo startapp blog "$tmpdir/compare/apps/blog"
```

Compare the temporary output against the real project:

```bash
diff -ru "$tmpdir/compare" /path/to/current/project
```

Review differences in:

- `go.mod`
- `.env.example`
- `.gitignore`
- settings packages
- middleware setup
- URL routing
- admin registration
- queue setup
- deployment files
- generated app structure

Apply only the changes that match the project's local architecture. Keep local
secrets out of comparisons and do not copy `.env` values from any environment.

## Production Upgrade Checklist

- Target Gogo version is pinned in `go.mod`.
- `go mod tidy` is complete.
- `go test ./...` passes.
- `make ci` passes when the generated project includes the template Makefile.
- `gogo makemigrations --check --dry-run` is reviewed.
- `gogo migrate --plan` is reviewed.
- `gogo showmigrations` is captured before and after migration.
- Database backup and restore test are complete.
- Static files are collected for the new version.
- Workers and beat are upgraded with the same application version as the web
  process.
- Queue brokers and result backends are compatible with the target version.
- Admin login, object add/change/delete, auth login/logout, password reset,
  API endpoints, media upload, and background tasks are smoke-tested.
- Rollback instructions include application version, migration posture, queue
  worker version, and static assets.

## Rollback Rules

Rollback depends on whether migrations changed data or schema.

- If no migrations were applied, roll back the application binary and static
  files.
- If only additive migrations were applied, rolling back the application may be
  safe when the old version ignores the new columns or tables.
- If destructive or data-transforming migrations were applied, restore from
  backup or run a reviewed reverse migration only when it is explicitly safe.
- Stop workers before rolling back queue message format changes.
- Drain or isolate queues when old workers cannot read new message envelopes.

When rollback is uncertain, keep the new application disabled and restore the
database from the verified backup instead of guessing.

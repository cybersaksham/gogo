The queue package provides Celery-style task registration, signatures,
envelopes, brokers, result backends, retries, worker execution, beat schedules,
routing, rate limits, events, inspection, security, admin metadata, and canvas
primitives.

## Register a task

```go
package docs

import (
	"context"

	"github.com/cybersaksham/gogo/queue"
)

func RegisterTask(app *queue.App) error {
	_, err := app.RegisterTask("blog.publish", func(context.Context, ...any) (any, error) {
		return "ok", nil
	}, queue.TaskOptions{Queue: "default"})
	return err
}
```

## Run queue processes

```bash
go run manage.go worker
go run manage.go beat
go run manage.go inspect
go run manage.go queues
```

## Reliability settings

Configure retries, retry delays, backoff, jitter, soft and hard timeouts, queue,
routing key, priority, acknowledgement policy, result handling, rate limits, and
idempotency expectations explicitly for production tasks.

## Brokers and backends

`memory` and `memory://` broker, backend, and schedule-store URLs are local
development and test runtimes. Production deploy checks reject memory broker and
result backend URLs.

`redis://` and `rediss://` broker and result backend URLs use Redis. Redis
brokers preserve task envelope metadata, delayed tasks, in-flight deliveries,
ack/nack/requeue behavior, and visibility-timeout recovery after worker exits.
Redis result backends store terminal state, child task IDs, group metadata, and
chord counters so separate processes can wait for task results.

`amqp://` and `amqps://` are unsupported runtime URLs until a real RabbitMQ
transport is registered; they do not fall back to memory.

Validate Redis worker configuration before deploying:

```bash
go run manage.go worker --check \
  --broker-url "$GOGO_BROKER_URL" \
  --result-backend "$GOGO_RESULT_BACKEND"
```

Canvas workflows live under `queue/canvas` and include chains, groups, chords,
chunks, maps, and serialized workflow types.

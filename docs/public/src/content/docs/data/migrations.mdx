---
title: Migrations
description: Generate, inspect, apply, squash, and optimize database schema changes.
---

Migrations live in each app's `migrations/` directory and describe schema
changes with explicit operations.

## Main workflow

```bash
go run manage.go makemigrations
go run manage.go sqlmigrate blog 0001_initial
go run manage.go migrate
go run manage.go showmigrations
```

## Maintenance commands

```bash
go run manage.go squashmigrations blog 0001 0005
go run manage.go optimizemigration blog 0005
```

## Rules

- Add migrations whenever model metadata changes database schema.
- Review generated operations before applying destructive changes.
- Keep data migrations idempotent and reversible where possible.
- Prefer additive schema changes in production systems.
- Do not edit already-applied migrations unless the project is private and the task explicitly asks for migration history cleanup.

Migration SQL rendering uses the configured dialect so teams can review the
exact SQL before rollout.

Applied migration history is recorded in `gogo_migrations` with dialect-aware
SQL. PostgreSQL history queries use `$1` placeholders, SQLite uses `?`, and
history upserts use `ON CONFLICT(app, name) DO UPDATE` for consistent behavior
across supported databases.

`go run manage.go migrate` takes a database-backed lock in
`gogo_migration_lock` before applying schema or history changes. A concurrent
migration run fails before operations execute; `migrate --plan` remains
read-only and does not take the lock.

`migrate --fake-initial` is for baselining an existing schema. It records an
initial migration without running SQL only when all tables declared by that
initial migration already exist; otherwise the migration is applied normally.
